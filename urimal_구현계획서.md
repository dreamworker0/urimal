## 1. 프로젝트 개요

### 목적

사회복지사가 계획서·주간업무보고서 등 실무 문서를 작성할 때  
**바르고 자연스러운 우리말**로 윤문할 수 있도록 돕는 도구를 만든다.

### 핵심 원칙 (이 프로그램의 목표)

1. **주술 호응 확보**: 주어와 서술어의 관계를 명확히 하여 문장의 기본 틀을 바로잡습니다.
2. **논리적 일관성 구축**: 문장 간의 연결 고리를 자연스럽게 만들어 전체 글의 흐름과 설득력을 높입니다.
3. **실용적 어휘 및 조사 사용**: 사회복지 현장에 맞는 실용적인 어휘를 선택하고, 우리말다운 자연스러운 조사를 활용합니다.

### 기반 자료

한덕연 선생님의 「우리말 공부」 — 사회복지 실무 현장 오류 36개 항목

## 3. Phase 2: 윤문 학습 프로그램 (웹 애플리케이션)

### 목표

사회복지사가 HWP·DOCX 파일을 업로드하면 **왜 틀렸는지 이해하며 스스로 교정할 수 있는 학습형 웹 도구**를 제공한다.  
단순 결과물 변환이 아니라, 우리말 오류 패턴과 한덕연 근거를 함께 보여줘 실력이 쌓이는 경험을 제공한다.

---

### 3-1. 확정된 기술 스택

| 레이어 | 기술 | 플랫폼 | 비용 |
|---|---|---|---|
| **프론트엔드** | Vite + React (TypeScript) | Vercel (예정) | 무료 |
| **백엔드** | Node.js + Express | Render.com (Starter 예정) | $7/월 |
| **HWP/HWPX 렌더링** | `@rhwp/core` 라이브러리 | 브라우저 (SVG 렌더링) | - |
| **문서 파싱 및 분석**| `kordoc` CLI + Gemini API | 백엔드 내 실행 | - |
| **AI 파이프라인** | Gemini API (`gemini-3.1-pro-preview` 등) | 서버 환경변수 키 관리 | 사용량 기반 |
| **DB / 스토리지** | **없음** | 메모리 처리 후 즉시 폐기 | - |

> **kordoc**이 Node.js CLI 도구이므로 Python 백엔드 불필요. 언어를 JS/TS로 단일화.  
> **파일 무보관 원칙**: 업로드 파일은 처리 중 메모리에만 존재하고 분석 후 즉시 폐기.  
> **지원 확장자**: 현재 HWP, HWPX 지원 (DOCX, PDF는 추후 지원 예정).

### 3-2. 전체 아키텍처

```
[사용자 브라우저]
       ↕
[Vite + React] ─────────────────── Vercel
  파일 업로드 UI
  학습형 교정 인터페이스
  결과 다운로드
       ↕ REST API (JSON)
[Node.js + Express] ────────────── 백엔드 서버
  ├── kordoc CLI
  │     HWP / HWPX 텍스트 파싱
  │     → 구조화 객체 (prose[], tables[])
  ├── Gemini API 호출
  │     im-not-ai 파이프라인 적용
  │     → 오류 탐지 JSON + 수정 제안
  └── 응답 반환 (파일 저장 없음 ✅)
        {
          segments: [...],   ← 원문 단락 + 오류 위치
          errors: [...],     ← 오류 목록 + 수정 + 근거
          rewritten: "..."   ← 최종 윤문 전문
        }
```

### 3-4. UI 설계

#### 주요 화면: 인라인 문서 뷰어 (Inline Document View)

기존 2단 Diff 뷰 방식에서 **원본 문서의 레이아웃을 유지하는 인라인 뷰어 방식**으로 진화.
`@rhwp/core`를 사용해 HWP/HWPX를 SVG로 렌더링하고, 그 위에 AI가 찾아낸 오류 영역을 하이라이트(툴팁) 처리.

```
┌────────────────────────────────────────────────────────┐
│  [원본 HWP 레이아웃 유지 영역]                           │
│                                                        │
│  저희 기관은 어르신들에게 행복한 시간을 [만들어 주어] ───┐ │
│  진심으로 고맙습니다.                   [AI 교정 제안]   │ │
│                                       "만들어"로 수정   │ │
│                                       (해 주다 과잉)    │ │
│                                       [적용] [건너뛰기]  │ │
└────────────────────────────────────────────────────────┘
```

- 원문: `@rhwp/core`를 통한 HWP 원본 SVG 렌더링
- 인터랙션: 오류가 발견된 텍스트 위치에 오버레이 하이라이트 적용.
- 클릭 시 수정 제안 및 우리말 근거(한덕연 항목) 툴팁/팝업 제공.

#### 부가 탭: 오류 카드 (학습 요소)

```
[Diff 뷰] [오류 목록]  ← 탭 전환
```

오류 목록 탭에서 SW-01~12 카테고리별 오류 카드 제공:

| 항목 | 내용 |
|---|---|
| **카테고리 배지** | SW-01~12 색상 구분 |
| **원문 표현** | 원문 그대로 |
| **수정 제안** | Gemini 파이프라인 출력 |
| **우리말 근거** | 한덕연 항목 번호 + 핵심 설명 |

## 7. 운영 거버넌스 (Rules & Workflow)

이 프로젝트가 현장에서 신뢰받고 꾸준히 발전하기 위한 운영 원칙과 프로세스입니다.

### 7-1. 운영 룰 (Rules & Guidelines)
1. **무보관 및 개인정보 보호 (Stateless & Privacy First)**
   - 사회복지 문서는 민감 정보(클라이언트 이름, 주소, 병력 등)가 포함되므로 절대 DB나 스토리지에 저장하지 않습니다.
   - 분석 및 교정 처리 후 메모리에서 즉시 폐기합니다.
2. **원문 훼손 금지 (Fidelity First & No Over-Polish)**
   - AI는 문학 작품을 쓰는 것이 아니라 오류를 교정하는 도구입니다.
   - 현장의 맥락이나 사회복지사의 의도를 해치는 과도한 윤문(Over-polish)을 지양하며, 교정 후에도 원문의 본질이 유지되도록 철저히 관리합니다.

### 7-2. 유지보수 워크플로우 (Workflow)
1. **사용자 피드백 루프**: 
   - 사용자가 "이 교정은 틀렸어요"라고 보고한 오작동 사례를 수집합니다 (원문 제외, 오류 패턴만).
   - 수집된 사례를 분석하여 `sw-rewriting-playbook.md` 업데이트 근거로 활용합니다.
2. **테스트 케이스 주도 프롬프트 관리 (TDD for Prompts)**: 
   - 플레이북을 수정하거나 AI 모델을 변경할 때, 기존 교정 기능이 망가지지 않도록(Regression 방지) `docs/test_cases/`에 축적된 샘플 텍스트를 이용해 먼저 자동 검증합니다.
3. **지속적 배포 (CI/CD)**: 
   - 프론트엔드(Vercel)와 백엔드(Render.com)는 `main` 브랜치 변경 시 자동 배포되도록 연결하여 빠른 개선을 지원합니다.

### 7-3. 향후 개발 자동화 스킬 (Future Skills)
- `urimal-tester`: 플레이북 갱신 시 의도치 않은 부작용(사이드 이펙트)이 없는지 다수의 테스트 문서로 빠르게 일괄 검증하는 파이프라인 무결성 검증 스크립트.
- `playbook-updater`: 새로운 오류 패턴 발견 시, 기존 `sw-tell-taxonomy.md` 구조에 맞춰 안전하게 마크다운을 업데이트해주는 자동화 스킬.
